# Perceptual UX Heuristics — AInvest Design System

> 本文档中的启发式原则是强制要求，不是可选的视觉润色——它们与设计 Token 规则具有同等权重，是核心设计要求。在开始每一个设计任务前必须重新阅读本文档，不得依赖记忆。这些原则规范的是元素之间的关系，确保每一个输出不仅符合设计规范，同时对真实的金融用户来说也直觉可用。

---

## 1. Visual Hierarchy — "第一眼落在哪里？"

**目标**：用户无需任何引导，在 3 秒内知道该看哪里。

**规则：**
- 每个页面最多只能有 **1–2 个主视觉焦点**，不能更多。如果所有内容都被强调，等于没有任何内容被强调
- 视觉权重由四个因素共同决定：**尺寸 + 颜色强度 + 位置 + 元素周围的留白**。调整其中任何一项都可以提升或降低层级
- 最重要的信息必须出现在**首屏 / 第一滚动区域**
- 次要内容应在视觉上更安静：更小的字号、次级文字颜色（`color-text-secondary`）、更少的周围留白

**设计完成前自查：**
- 你能识别出页面上视觉权重最高的单一元素吗？
- 如果用户只看 3 秒，他们能说出这个页面的主要用途吗？
- 是否有两个或更多元素在争夺相同的视觉权重？→ 其中一个必须让步

## 2. Reading Flow — "视线是否自然移动？"

**目标**：布局遵循人眼自然扫描的方向，用户无需费力就能吸收信息。

**规则：**
- 内容密集型页面（仪表盘、数据列表）默认采用 **F 型扫描**：强顶部横栏，然后向左侧垂直扫描
- 落地页/营销页默认采用 **Z 型扫描**：左上 → 右上 → 对角线 → 右下（CTA 所在位置）
- 信息流向为**从左到右、从上到下**。如果右上方区域未被使用，不得将高优先级内容放在左下方
- 避免"视线跳跃"：不要将相关信息分散放置，迫使视线横跨页面跳跃。相关内容应在空间上聚合
- **CTA 属于阅读路径的终点**，而非随意悬浮。Z 型页面的终点在右下角；F 型页面的终点在扫描自然结束的位置

**自查：**
- 从左上角向下追踪视线路径，它是连续流动的，还是会停顿？
- 主 CTA 是否位于用户读完价值主张后视线自然到达的位置？
- 是否有孤立元素打断了阅读节奏？

## 3. Interaction Affordance — "用户知道该做什么吗？"

**目标**：每个可交互元素无需标签或提示就能传达自身的可点击性/可操作性。

**规则：**
- 所有可交互元素必须具有可见的可操作性线索：与背景的对比差异、暗示可按压性的形状（按钮轮廓/填充），或悬停时的光标变化
- **CTA 文案必须使用动作动词**，而非名词或状态词。"Get Started"、"View Report"、"Add Position"——而非"Start"、"Reports"、"Addition"
- 不得让文字看起来像静态内容，而实际上是链接或触发器。如果它可以交互，它必须**看起来**可以交互（颜色、下划线，或明确的按钮处理）
- **Hover 和 Active 状态不是可选的**——它们是桌面端用户的主要可操作性信号。每个可交互元素都需要两者（按钮规范中已强制执行；对链接、卡片和列表项同样适用）
- 避免"神秘肉导航"——需要猜测的无标签图标。如果图标含义不是普遍共识，添加标签

**自查：**
- 遮住标签，你还能区分可交互元素和静态内容吗？
- 每个按钮都能回答"点击后会发生什么"这个问题吗？
- 是否有看起来可点击但实际不能点击的元素，或看起来是静态但实际可交互的元素？

## 4. Information Readability — "用户能毫不费力地吸收信息吗？"

**目标**：最小化认知负荷。用户是扫描，不是阅读。为扫描而设计。

**规则：**
- **始终分层文字**，使用定义好的文字角色：标题建立话题，正文传递内容，说明文字提供辅助细节。不得混用这些角色
- **行长度**：正文文字每行不超过约 70–80 个字符（14px 下约为 600–700px）。过长的行会迫使视线重新找到下一行的起点，打断阅读流
- **信息密度**：单个卡片或模块只传达一个概念。如果你发现自己在一个组件中堆叠了 4 个以上不同的数据点，考虑拆分或使用渐进式展示（展开/折叠、提示框、抽屉）
- **5 秒可扫描原则**：用户在任何区域只花 5 秒，就应该能提取到核心信息。如果不能，该区域内的层级结构已经失效
- 避免"文字墙"——使用留白、副标题或视觉分隔符在信息块之间创造呼吸空间

**自查：**
- 选取任意一个区域，只读标题和加粗文字，你理解这部分内容的要点吗？
- 是否有超过 3–4 行的文字块可以被拆分或概括？
- 每个卡片/模块是否只回答了用户的一个问题？

## 5. Color Semantics & Contrast — "颜色是在传递信息，还是只是装饰？"

**目标**：颜色传递含义，而非仅仅服务于美观。每一个颜色选择都是可解释的。

**规则：**
- **永远不要只用颜色作为唯一的含义信号**。始终搭配第二个信号：图标、标签、图案或位置。这同时服务于无障碍访问（色盲用户）和清晰度
  - ✅ 绿色上涨箭头 + 绿色 `+2.3%` 文字
  - ❌ 单独的绿色数字（色盲用户完全失去这个信号）
- **语义色彩只用于其对应功能**：success/up（`#009B67`）、error/down（#FF381A）、warning（`#FFAA00`）不得用于装饰目的。将错误红色用于非错误元素会破坏整个语义语言体系
- **对比层级**：页面上最重要的文字应具有最高对比度（primary text = 100% 不透明）。辅助文字使用 60%，次要辅助文字使用 40%。不得反转这个顺序
- CTA 按钮必须在页面背景中视觉突出。若品牌主色（`#165DFF`）在卡片背景上对比度不足，增加按钮视觉重量或使用轮廓——不得为按钮添加阴影
- WCAG AA 最低标准：正文文字 4.5:1，大文字（18px+ 加粗或 24px+ 常规）3:1

**自查：**
- 如果将设计去色（灰度），层级结构还能成立吗？
- 是否有任何语义色彩（success/error/warning）被用于它所描述功能之外的地方？
- CTA 是否无需寻找就能在视觉上突出显示？

## 6. Layout Stability & Structural Rhythm — "这个界面整体感觉是否连贯？"

**目标**：相同类型的页面感觉像一个家族。变化发生在内容层面，而非结构层面。

**规则：**
- **结构固定，内容可变。** 栅格、标题区域、导航位置、卡片布局——在同类型页面之间不得改变。只有其中的数据在变化
- **节奏来自一致的间距间隔。** 使用 8px 增量不只是合规规则——它创造了视觉节拍。打断它（例如在 16px/24px 上下文中出现 20px 间距）会产生潜意识的不适感
- **对齐即信任。** 逻辑上相关的元素必须共享一条对齐轴。即使内容正确，未对齐的元素也会传递混乱的信号
- 重复的结构模式降低认知负荷：一旦用户学会了一个卡片的使用方式，他们就能使用所有卡片。不要在没有明确理由的情况下引入结构上的新奇感
- 当布局必须改变时（不同内容类型、不同密度），用有意识的方式传递这种改变——使用不同的背景色、区域分隔线或容器——而不是一个看起来像失误的微妙偏移

**自查：**
- 并排对比同类型的两个页面，结构骨架是否完全一致？
- 是否有任何间距值在没有明确理由的情况下打断了 8px 节奏？
- 逻辑上相关的元素是否共享了视觉对齐轴？
